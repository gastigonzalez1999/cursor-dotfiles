# Write a PRD

## When to use

- Turning an idea into a product requirements document stakeholders can align on.
- Before significant build work when scope, success, and constraints are fuzzy.

## Steps

1. Capture **problem** and **target users** in one paragraph each; list non-goals explicitly.
2. State **goals** (measurable) and **non-functional** needs (performance, security, compliance, accessibility).
3. Describe **user stories** or **jobs-to-be-done** prioritized as P0/P1/P2.
4. Define **user-visible behavior**: flows, edge cases, empty/error states, permissions.
5. Specify **data** entities and integrations at a high level (sources, retention, PII).
6. List **open questions** and **assumptions**; mark what needs validation before build.
7. Add **acceptance criteria** testable by QA or automated checks where possible.
8. Add **rollout**: phases, feature flags, migration, and rollback.
9. Keep the doc concise; link to designs, APIs, or tickets instead of duplicating detail.
